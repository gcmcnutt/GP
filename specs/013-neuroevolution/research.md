# Research: 013-neuroevolution

## R1: Network Topology

**Decision**: Start with 14-16-8-3 (403 params), scale to 14-32-16-3 (1,011 params) if needed.

**Rationale**: Neuroflight (Koch et al., 2019) proved 6-32-32-4 (1,412 params) works for
real drone flight control on Cortex-M7. Our 14 inputs (more sensors) but 3 outputs (fewer
actuators) suggest a similar or smaller hidden layer suffices. The 14-16-8-3 baseline is
conservative — evolution will discover whether capacity is sufficient.

**Alternatives considered**:
- 14-8-3 (139 params): Too small — may lack capacity for cross-term computation
- 14-32-32-3 (1,635 params): Match Neuroflight size — viable but larger search space
- 14-64-32-16-3 (3,504 params): Overkill for 14 inputs, evolution may struggle

## R2: Activation Function

**Decision**: tanh everywhere (hidden + output layers).

**Rationale**: Output layer tanh maps directly to [-1, 1] control commands (SETPITCH,
SETROLL, SETTHROTTLE) with no clamping needed. Hidden tanh is zero-centered (unlike
sigmoid) and bounded (unlike ReLU, which causes dead neurons under mutation). Neuroflight
uses tanh. The neuroevolution literature consistently recommends tanh for control tasks.

**Alternatives considered**:
- ReLU: Faster to compute but unbounded output requires clamping; dead neurons harder to
  recover from without gradient signal
- Sigmoid: Non-zero-centered, introduces bias
- Evolved activations: Interesting research topic but adds complexity for marginal gain

## R3: Evolution Algorithm

**Decision**: Phase 1 uses simple GA (tournament selection, arithmetic crossover, Gaussian
mutation). Switch to sep-CMA-ES if GA plateaus.

**Rationale**: Simple GA reuses existing GPc++ tournament infrastructure with minimal
changes. Uber (2017) proved it works at 4M+ parameters. For 403 params it's trivially
sufficient. CMA-ES is more sample-efficient but requires new infrastructure.

**Alternatives considered**:
- Full CMA-ES: O(n^2) memory at 403 dims — impractical
- sep-CMA-ES: Excellent sample efficiency (pop ~22), practical at 403 dims, but more
  complex to implement. Reserve for Phase 2 if GA plateaus.
- OpenAI ES: Needs massive parallelism (720+ cores), impractical at 8 threads
- NEAT: Unnecessary — we have a known-good fixed topology

## R4: Embedded Inference Framework

**Decision**: Hand-written C forward pass, generated by nn2cpp tool.

**Rationale**: For 403 params, a hand-written forward pass is ~50 lines of C with zero
dependencies. This matches the existing bytecode2cpp pattern exactly. Framework overhead
(TFLM: 10-26 KB, Edge Impulse: 15 KB) is unjustified for a trivial computation. The
existing sin/cos/atan LUT pattern in gp_evaluator_portable.cc provides the template for
a tanh LUT.

**Alternatives considered**:
- TFLM: 10-26 KB flash overhead, complex build integration, overkill
- NNoM: Good upgrade path for future larger networks, but unnecessary now
- nn4mc: Generates C from Keras — adds Python dependency, unnecessary
- CMSIS-NN: INT8 SIMD gives 2x throughput but network is so small (~15 µs) it doesn't matter
- Edge Impulse: Proven on nRF52840 (1 ms inference) but adds vendor dependency

## R5: Numeric Precision

**Decision**: FP32 throughout (compute and storage).

**Rationale**: Cortex-M4F has native FP32 FPU (1-cycle MAC via VMLA.F32). No native FP16
compute. INT8 SIMD via CMSIS-NN gives 2x throughput but the network is so small (~15 µs
at FP32) that optimization is premature. 403 weights × 4 bytes = 1.6 KB — negligible vs
256 KB RAM.

**Alternatives considered**:
- FP16 storage + FP32 compute: Halves weight storage (VCVT.F32.F16 is 1 cycle) but
  adds conversion complexity for negligible savings
- INT8 quantized: 4x storage reduction, 2x throughput via SIMD, but quantization noise
  is more impactful on small networks and adds calibration complexity

## R6: Serialization Format

**Decision**: New binary format with magic number. Intentionally incompatible with GP tree
and bytecode formats. No backward compatibility required.

**Rationale**: This is a fork in the road. NN genomes are flat weight arrays — fundamentally
different from GP trees or bytecode programs. A new format allows clean design without
legacy constraints. The existing minisim format detection (first-bytes heuristic) extends
naturally with a magic number prefix.

**Format sketch**:
```
Magic: 4 bytes ("NN01")
Topology: num_layers (uint32) + layer_sizes[] (uint32 × num_layers)
Weights: float32[] (flattened, layer-major order)
Metadata: fitness (float64), generation (uint32), S3 key (string)
```

## R7: Eval Pipeline Unification Strategy

**Decision**: Extract shared code into standalone modules (fitness_computer, eval_logger),
define ControllerBackend interface, refactor both existing eval paths to use it.

**Rationale**: The current codebase has ~350 lines duplicated between MyGP::evalTask() and
BytecodeEvaluationGP::evalTask(). Adding NN as a third path without unification would
triple the maintenance burden. The ControllerBackend interface is minimal:
`evaluate(AircraftState&, PathProvider&) → void` (sets control commands as side effect).

**Key insight**: The sensor functions in gp_evaluator_portable.cc are already factored out
and shared. The duplication is in the fitness computation loop, scenario handling, and
logging — not in the controller evaluation itself.

## R8: Population Sizing

**Decision**: Start with pop=500 (matching current GP runs). Reduce to ~50 with CMA-ES
if per-generation time is too long.

**Rationale**: Current GP runs: pop=500, 8 threads, ~160s/gen. NN evolution has identical
per-individual eval cost (same simulation). So pop=500 gives same ~160s/gen. This is
acceptable for initial experiments. If we need faster iteration, CMA-ES at pop=22 gives
~8 min/gen (8 threads). GPU-native (011) future path enables pop=10K+ at similar speed.

## R9: Sensor Input Vector

**Decision**: 14 inputs using existing sensor functions. No new sensors needed.

**Rationale**: The GP already has GETDPHI (lateral bearing), GETDTHETA (vertical bearing),
GETDIST (range), attitude sensors (roll_rad, pitch_rad), and rate sensors. These encode:
- Cross-track position: sin(dPhi) × dist, sin(dTheta) × dist (computable by hidden layer)
- Gravity direction: implicit in roll_rad + pitch_rad
- Temporal dynamics: explicit via _RATE inputs
- Current control state: roll/pitch/throttle commands (feedback)

Adding cross-track sensors (GETCROSS_LAT, GETCROSS_VERT) would duplicate information
already available via angular sensors × distance. The NN can learn these cross-terms
in one hidden layer — that's the whole point of switching from GP to NN.

## R10: xiao-gp Code Generation Pattern

**Decision**: nn2cpp generates `nn_program_generated.cpp` with identical function signature
to `generatedGPProgram()`. Build selects GP or NN via file inclusion.

**Rationale**: The existing xiao-gp build includes `generated/gp_program_generated.cpp`
which provides `generatedGPProgram(PathProvider&, AircraftState&, gp_scalar)`. The NN
equivalent provides `generatedNNProgram()` with the same signature. The build system
(PlatformIO) selects which file to include. No changes to `msplink.cpp` dispatch logic
beyond swapping the function name (or using a function pointer configured at build time).

The generated code embeds weights as `static const float[]` in flash, unrolls the
layer loops for the specific topology, and calls `fast_tanh()` from the portable math
module. Output is a standalone .cpp file with no dependencies beyond `aircraft_state.h`
and `gp_evaluator_portable.h` (for sensor functions).
