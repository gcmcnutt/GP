# Implementation Plan: 013-neuroevolution

**Branch**: `013-neuroevolution` | **Date**: 2026-03-12 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/013-neuroevolution/spec.md`

## Summary

Replace tree-based GP controller with evolved fixed-topology neural network. Absorbs
007-unify-eval as Phase 1 (shared evaluation backend interface). Implements NN forward
pass, weight evolution via GA/CMA-ES, new serialization format, and embedded code
generation for xiao-gp deployment. GPU-native evaluation (011) is a future scaling path.

## Technical Context

**Language/Version**: C++17 (g++, CMake 3.10+)
**Primary Dependencies**: Eigen3 (vectors/quaternions), Boost (serialization, logging, threads), GoogleTest 1.14.0
**Storage**: S3 for evolution artifacts (new NN archive format, incompatible with GP archives)
**Testing**: GoogleTest 1.14.0 — unit tests for forward pass, serialization round-trip, evolution operators
**Target Platform**: Linux (training), nRF52840 ARM Cortex-M4F @ 64 MHz (deployment)
**Project Type**: Evolution engine (autoc) + embedded firmware (xiao-gp)
**Performance Goals**: NN inference <1ms on nRF52840; training throughput >= current GP rate (~160s/gen at pop=500)
**Constraints**: 256KB RAM / 1MB Flash on embedded; FP32 native FPU; 100ms control loop (10 Hz)
**Scale/Scope**: 403-1,635 NN weights; pop 22-500 (CPU) scaling to 10K+ (GPU future)

### I. Testing-First

**Status**: PASS

- Phase 1 (unify eval) verified by existing dual-mode parity tests
- Phase 4 (evolution) tested by convergence on known-easy problem before full sim

### II. Build Stability

**Status**: PASS with justification

- GP tree path remains functional throughout. NN path is additive.
- Multi-repo impact: xiao-gp gets new `nn_evaluator_portable` and `nn_program_generated.cpp`
  in Phase 5 — build verified on all three repos at that point.
- CRRCSim: no changes needed (minisim handles NN format detection internally).

### III. Dual-Mode Parity

**Status**: JUSTIFIED EXEMPTION

The dual-mode parity principle requires GP tree and bytecode to produce identical results.
NN evaluation is a **third mode** with intentionally different behavior — it's not a
reimplementation of GP tree evaluation. However:

- The NN evaluation mode MUST produce deterministic results for the same weights + inputs
- Desktop and embedded NN inference MUST produce bit-exact results (same LUT, same FP32)
- Fitness computation is shared (Phase 1 unified eval) ensuring identical scoring

The spirit of dual-mode parity (deployed behavior matches trained behavior) is preserved:
NN weights trained on desktop produce identical control outputs on embedded hardware.
